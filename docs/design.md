# adiFit — Review-Based Fit, Usage, and VOC Intelligence

- **한 줄 컨셉:** 리뷰에서 사이즈·착용감·색상/디자인·사용 용도를 구조화해 고객에게는 **근거 기반 핏 어드바이스**를, 머천다이저에게는 **VOC 세그먼트와 실행 액션**을 제공한다.

---

## 1. Why This Design

많은 리뷰 분석 도구는 요약이나 키워드 추출에서 멈춘다. adiFit은 리뷰를 실제 의사결정에 사용할 수 있도록 구조화하고, 소비자 경험과 내부 운영 액션으로 연결한다.

| 레버 | 구현 위치 |
|---|---|
| **엔지니어링 깊이** | 속성 스키마 · 구조화 출력 · RAG 근거 통제 · embedding clustering · 검증(eval) 레이어 |
| **아디다스 특화** | 사이즈/핏 · 착용감 · 용도 · 디자인/색상 · 신발 도메인 · 반품/PDP 액션 |
| **비즈니스 번역** | 대시보드 액션 패널 · KPI(반품률·전환·CS 절감) — 기술경영 강점 |

하나의 엔진, 두 개의 얼굴 → **소비자 경험(옴니·개인화)** 과 **내부 운영(반품↓)** 을 동시에 커버.

---

## 2. 목표 / 비목표

**목표 (Working core — 반드시 실제 동작)**
- 리뷰 → 속성기반 감성·이슈 추출(구조화 출력)
- 상품별 집계 + "사이즈 신호"(작음/정사이즈/큼) 정규화
- 사용 용도(러닝/일상/출퇴근/대회), 착용감, 디자인/색상 태그 추출
- 리뷰 임베딩 기반 고객 세그먼트 클러스터링 + 월별 VOC 트렌드
- 리뷰 근거 기반 핏 어드바이저 답변(RAG, 출처 인용, 근거 밖 주장 금지)
- 머천다이저 대시보드(차트 + 액션 패널)
- 경량 검증: 추출 정확도 + 답변 근거성

**비목표 (개념/Mock 허용 — 슬라이드에서 명시)**
- 프로덕션 PDP 임베드/실서비스 인증·배포 인프라
- 대규모 실시간 파이프라인, A/B 테스트 실측
- 완전 자동 데이터 수집(스크래핑) — 라이선스 이슈 회피 위해 직접 정의한 데모 리뷰 사용

---

## 3. 사용자 & 시나리오

- **소비자 (얼굴 B, 헤드라인):** "나 나이키 265에 발볼 넓은데 울트라부스트 몇 신어야 해?" → 관련 리뷰 검색 → **사이즈 추천 + 근거 인용 + 확신도**. 근거 부족하면 "리뷰로는 판단 어려움" 명시.
- **머천다이저 (얼굴 A, 근거):** 특정 상품의 속성별 감성·사이즈 편향·핵심 이슈·대표 인용을 한눈에 → **"PDP에 '반 사이즈 업' 안내 추가 → 반품 위험 플래그"** 같은 실행 제안.

---

## 4. 아키텍처

```
리뷰 데이터(아디다스 데모 큐레이션)
   │
   ▼  [코어 엔진 · 실제 구현]
① 속성기반 추출 (Gemini structured output)
     핏/사이즈 · 품질 · 착용감 · 내구성 · 배송 · 디자인
     + 속성별 감성(pos/neu/neg) + 사이즈 신호(runs_small/true/runs_large)
     + 사용 용도 · 착용감 · 디자인/색상 · 고객 맥락 · VOC 이슈 태그
   │
② 집계 → 상품별 속성 감성 분포, 사이즈 편향률, 이슈 랭킹, 대표 인용
   │
③ 색인(RAG) → Gemini 임베딩 → 인메모리 코사인 검색
   │
   ├─▶ [얼굴 B] 핏 어드바이저(소비자)
   │      질의 → top-k 리뷰 검색 → Gemini 근거기반 답변(출처 id 인용, 확신도)
   │      가드레일: 검색 근거 밖 사이즈 주장 금지, 근거 부족 시 "모름"
   │
   ├─▶ [얼굴 A] 머천다이저 대시보드(내부)
          속성 감성 차트 · 사이즈 편향 게이지 · 이슈 랭킹 · 대표 인용 · 액션 패널

   └─▶ [연구 강점 레이어] 세그먼트 & 트렌드
          embedding clustering · 사용 맥락 세그먼트 · 월별 VOC issue trend

[검증 레이어] 손라벨 골드셋(24건) → 사이즈 신호 정확도 + 어드바이저 근거성(출처 커버리지)
```

## 5. 컴포넌트 (각각: 역할 / 인터페이스 / 의존)

1. **data loader** — 역할: 아디다스 데모 리뷰 로드, 정규화(product_id, rating, text, date). 인터페이스: `load_reviews() -> list[dict]`. 의존: 로컬 JSON.
2. **aspect extractor** — 역할: 리뷰 1건 → 속성별 감성 + 사이즈 신호 + 사용 맥락/VOC 태그(구조화). 인터페이스: `extract(review_text) -> ReviewExtraction`. 의존: Gemini structured output. 캐시로 재호출 최소화.
3. **aggregator** — 역할: 상품 단위 집계(감성 분포·사이즈 편향·이슈·대표 인용). 인터페이스: `aggregate(product_id) -> ProductInsight`. 의존: extractor 출력.
4. **retriever (RAG)** — 역할: 리뷰 임베딩·색인·검색. 인터페이스: `search(query, product_id, k) -> [Review]`. 의존: 임베딩 모델 + 벡터 스토어.
5. **fit advisor** — 역할: 질의 → 검색 → 근거기반 답변(가드레일). 인터페이스: `advise(query, user_ctx) -> AdviceWithCitations`. 의존: retriever + Gemini API.
6. **dashboard UI (Streamlit 탭 A)** — 역할: ProductInsight 시각화 + 액션 패널. 의존: aggregator, Plotly/Altair.
7. **advisor UI (Streamlit 탭 B)** — 역할: 채팅형 입력/답변/출처 표시. 의존: fit advisor.
8. **segments** — 역할: embedding clustering + 태그 기반 세그먼트 라벨링 + 월별 VOC trend. 의존: retriever vectors, extraction tags.
9. **evaluator** — 역할: 골드셋 대비 정확도 + 답변 근거성 측정, 리포트 출력. 인터페이스: `run_eval() -> EvalReport`. 의존: extractor, advisor, 골드셋.

각 컴포넌트는 독립 테스트 가능하도록 순수 함수 + 명확한 입출력 유지. 파일이 커지면 분리.

## 6. Data

- **현재 구현:** 아디다스 대표 상품 3종의 리뷰 24건을 직접 정의한 데모 데이터.
- **운영 확장:** 실제 리뷰 수천~수만 건을 연결하면 동일한 스키마로 세그먼트 drift와 VOC spike를 추적 가능.

---

## 7. 기술 스택

- **언어/앱:** Python + **Streamlit**(탭 2개) — 데모 녹화·스크린샷 최적
- **LLM:** **Gemini API** — 추출·생성 모두 structured output
- **RAG:** Gemini embedding + 인메모리 코사인 검색(경량)
- **클러스터링/시계열:** scikit-learn KMeans/PCA + 월별 태그 집계
- **시각화:** Plotly, pandas
- LLM 구조화 추출과 시계열·클러스터링 연구 경험을 함께 보여주는 구성

---

## 8. Risks & Mitigation

- **RAG 환각/근거 밖 주장** → 가드레일(검색 근거만 사용, 부족 시 "모름"), 출처 id 강제 인용, 근거성 eval로 수치 증빙.
- **LLM 호출 비용/시간** → 추출 결과 캐시, 데모셋 소량 유지.
- **데이터 라이선스** → 직접 정의한 데모 리뷰 사용으로 외부 리뷰 저작권 리스크 회피. 실제 운영 데이터 연결은 확장 방향으로 설명.
- **범위 확장 리스크** → 세그먼트/시계열은 데모 샘플 기준으로 구현하고, 통계적 유의성은 운영 규모 데이터에서 확보한다고 명시.
